# 이모지월드 (EmojiWorld) 향후 백엔드 도입 및 보안 구현 계획

현재 배포된 버전은 정적 HTML + localStorage 기반의 클라이언트 사이드 앱입니다. 이 문서는 향후 "완전 배포(프론트엔드 + 백엔드 연동)" 시점에 발생할 수 있는 보안 취약점과 이를 어떻게 해결하고 구현할 것인지 4단계 핵심 진화 로드맵을 기록합니다.

---

## 🚀 1단계 진화: 데이터를 중앙에 저장하기 (DB 연동)

### 목표

- `localStorage`에 저장하던 닉네임, 좋아요 개수, 포스트 데이터 등을 중앙 데이터베이스(예: MySQL, PostgreSQL, Firebase)로 치환.

### 🔴 보안 위협: SQL Injection

해커가 닉네임이나 검색어 등 사용자 입력 필드를 통해 악성 SQL 명령어를 서버로 주입하여, 전체 회원 정보를 유출하거나 데이터를 고의로 삭제할 수 있습니다.

### 🛡️ 보안 구현 (수정 계획)

- **Parameterized Queries (Prepared Statements) 사용:** 모든 백엔드 언어/프레임워크(Node.js, Spring 등)에서 지원하는 쿼리 매개변수화 기법을 사용하여, 사용자의 입력값이 SQL 명령어가 아닌 단순 '데이터'로만 취급되도록 강제합니다.
- **ORM 도입:** TypeORM, Prisma, JPA 등 안전한 ORM 라이브러리를 사용하여 쿼리를 처리합니다.

---

## 🚀 2단계 진화: 진짜 로그인 시스템 만들기 (인증/보안)

### 목표

- 이메일/비밀번호 혹은 소셜 로그인(Google, Kakao 등)을 연동하여 확실하고 유일한 유저 식별(Authentication) 시스템 도입.

### 🔴 보안 위협: CSRF (Cross-Site Request Forgery) & Session Hijacking

해커가 사용자가 로그인된 상태를 교묘하게 이용하여, 사용자가 의도하지 않은 요청(예: 악성 포스트 작성, 정보 변경 등)을 강제로 전송하게 만들 수 있습니다.

### 🛡️ 보안 구현 (수정 계획)

- **CSRF Token 도입 또는 SameSite 속성 적용:** 세션 기반 인증일 경우 토큰을 클라이언트에 발급하고 검증하며, 쿠키에는 반드시 `SameSite=Lax`나 `Strict`를 설정하여 외부 사이트에서의 요청을 차단합니다.
- **Secure 및 HttpOnly 설정:** 쿠키에 세션 ID를 담을 때 스크립트에서 접근하지 못하도록 `HttpOnly` 속성을 부여합니다.
- **토큰 기반 인증 (JWT):** 만약 API 기반으로 분리한다면, 엑세스 토큰(Access Token)은 메모리에, 리프레시 토큰(Refresh Token)은 Secure + HttpOnly 쿠키로 관리합니다.

---

## 🚀 3단계 진화: 진짜 사진 파일 서버에 업로드하기

### 목표

- 사용자가 업로드하는 프로필 및 피드 이미지(Data URL/Base64) 대신 실제 이미지 파일(`.jpg`, `.png`, `.webp`)을 AWS S3 등 클라우드 스토리지에 업로드하고 서빙하기.

### 🔴 보안 위협: 악성 파일(웹셸) 업로드

해커가 확장자만 이미지로 바꾼 악성 프로그램(실행 파일, `.php`, `.jsp` 등)을 서버에 업로드하여, 해당 파일을 실행시킴으로써 백엔드 서버 권한을 완전히 탈취할 수 있습니다.

### 🛡️ 보안 구현 (수정 계획)

- **엄격한 매직 바이트(Magic Byte) 검증:** 확장자나 프론트엔드의 MIME 타입에 의존하지 않고, 백엔드 서버에서 파일의 실제 헤더(내부 구조)를 읽어 진짜 이미지인지 검증합니다.
- **이미지 재인코딩(Re-encoding) 및 리사이징:** 백엔드에서 원본 이미지를 받아 그래픽 라이브러리(Sharp, ImageMagick 등)로 안전하게 다시 그립니다. 악성 코드가 섞여 있어도 이 과정에서 제거됩니다.
- **실행 권한 박탈:** 파일이 저장되는 S3 등 저장소 폴더에는 어떠한 경우에도 코드 실행 권한을 부여하지 않습니다.

---

## 🚀 4단계 진화: 사용자 간 소통 및 권한 설계

### 목표

- 다른 사람의 글이 실시간으로 피드에 보이고, 내 포스트에 다른 사람의 좋아요와 댓글이 달리게 하는 등 상호작용 기능 도입.

### 🔴 보안 위협: IDOR (Insecure Direct Object References) - 불완전한 접근 제어

해커가 클라이언트의 네트워크 요청 파라미터(예: `delete_post_id=10`)를 남의 글 번호(예: `11`)로 무단 변조할 경우, 권한 없는 자에 의해 포스트가 삭제되거나 수정될 수 있습니다.

### 🛡️ 보안 구현 (수정 계획)

- **소유권 및 권한 엄격 검증:** 요청된 글 번호만 보고 삭제/수정하는 것이 아니라, 해당 요청을 보낸 사용자의 세션 정보와 실제 그 글의 작성자 ID가 정확히 일치하는지 백엔드 컨트롤러 층에서 무조건 검증합니다.
- **유추하기 어려운 자원 ID (UUID 활용):** 게시글 번호를 `1, 2, 3`처럼 순차적으로 부여하지 않고, 예측 불가능한 `UUID`로 관리하여 해커가 남의 글 번호를 찍어 맞추는 것을 어렵게 만듭니다.

---

> 이 플랜은 향후 정식 프로덕션(프론트+백 연동) 빌드 시 적용합니다. 현재 브라우저 기반의 정적(Local) 데모 앱에서는 프론트엔드 보안(CSP, 입력 Sanitization 등)이 견고하게 조치되어 있어 안전하게 쇼케이스가 가능합니다.
